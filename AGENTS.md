# Repository Guidelines

## Project Structure & Module Organization

This is a PHP 8.4 library for reading and writing spreadsheets. Source code lives in `src/` under the PSR-4 namespace `CarlosChininin\Spreadsheet\`.

- `src/Reader/` contains reader contracts, options, traits, and OpenSpout/PhpSpreadsheet implementations.
- `src/Writer/` contains writer contracts, options, traits, and backend-specific writers.
- `src/Shared/` contains reusable helpers, formats, colors, ranges, and spreadsheet type constants.
- `tests/` is reserved for dev tests under `CarlosChininin\Spreadsheet\Tests\`.
- `vendor/` is generated by Composer and must not be edited.

## Build, Test, and Development Commands

```bash
composer install
```
Installs runtime and dev dependencies from `composer.lock`.

```bash
composer validate --strict
```
Validates package metadata before publishing or opening a PR.

```bash
vendor/bin/php-cs-fixer fix --dry-run --diff
```
Checks formatting without changing files.

```bash
vendor/bin/php-cs-fixer fix
```
Applies the project coding standard.

```bash
composer dump-autoload
```
Regenerates autoload metadata after adding classes or namespaces.

## Coding Style & Naming Conventions

Use strict PHP with `declare(strict_types=1);` and the PIDIA file header configured in `.php-cs-fixer.dist.php`. Follow Symfony PHP-CS-Fixer rules, including risky rules. Use 4-space indentation, typed properties, constructor property promotion where useful, and explicit return types.

Class names should match their file names and namespaces, for example `src/Writer/OpenSpout/SpreadsheetWriter.php` => `CarlosChininin\Spreadsheet\Writer\OpenSpout\SpreadsheetWriter`. Keep backend-specific code inside its backend folder and shared logic in traits or `src/Shared/`.

## Testing Guidelines

No PHPUnit configuration is currently committed. When adding tests, place them in `tests/` using the dev namespace and name files `*Test.php`. Prefer small fixtures that cover XLSX, CSV, and ODS behavior without committing generated vendor or temporary files. Until a runner is added, verify changes with Composer validation, PHP-CS-Fixer dry run, and targeted manual scripts using `vendor/autoload.php`.

## Commit & Pull Request Guidelines

Recent history uses short imperative messages such as `fix lint code` and `update phpspreadsheet v5`. Keep commits focused and describe the changed behavior or dependency update.

Pull requests should include: a brief summary, affected reader/writer backend, validation commands run, linked issue when available, and notes about compatibility with OpenSpout, PhpSpreadsheet, Symfony, or PHP versions.

## Agent-Specific Instructions

Do not edit `vendor/`. Avoid broad rewrites; preserve public interfaces in `ReaderInterface`, `WriterInterface`, `ReaderOptions`, and `WriterOptions` unless the task explicitly asks for a breaking change.
